## 2. 자동 검증 환경의 한계

자동 테스트는 다음 항목을 다루지 **못한다** — 모두 사용자 PC 에서 수동 확인 대상.

- **실제 GPU 환경의 throughput 향상치** — 단위 테스트는 카운터로 동시성 cap 만 검증한다. RTX 3090 등 실측 PC 에서 같은 데이터셋을 M2 대비 얼마나 빨리 끝내는지는 §3 에서 확인.
- **Ollama 서버의 실제 동시 호출 거동** — respx 로 HTTP 만 모킹된다. 같은 `gemma4:e4b` 모델이 backend 에서 정말 OOM 없이 2개를 동시에 처리하는지(또는 단순히 큐잉만 하는지)는 실측.
- **CLIP forward 의 thread-safety** — `FakeBackend` 가 직렬화 invariant 만 검증. 실제 torch open_clip ViT-B/32 가 우리 lock 없이도 동작하는지(또는 정말 깨지는지)는 검증 안 함 — 보수적으로 lock 으로 직렬화.
- **250ms 디바운스의 체감 부드러움** — 단위 테스트는 호출 횟수 cap 만 본다. 실제 사용자 PC 에서 GUI lag/freeze 없이 부드럽게 진행되는지는 시각 확인.

### 3.3 throughput 비교 (선택, M2 대비)

같은 데이터셋(예: 위 10개 파일)을 M2 시점과 M2.1 시점에 각각 한 번씩 떨어뜨려 분석 완료까지의 wall clock 을 비교한다.

- 측정 방법: 라이브러리 탭의 `분석 상태` 컬럼이 모두 `ok` 또는 `partial` 이 될 때까지의 시간 측정.
- 기대치: RTX 3090 환경 기준 약 2~2.5x 단축. CPU only 환경에서는 효과가 작거나 미미(GIL + Ollama 단일 모델 큐잉 영향).
- M2 시점 측정치가 없다면 본 회 측정만 메모로 남겨도 충분.

### 3.4 안정성 — DB lock 에러 0건

분석이 한 차례 끝난 뒤:

```powershell
Select-String -Path $env:APPDATA\GameAssetHelper\logs\gah.log -Pattern "database is locked|OperationalError"
```

→ 0건이어야 한다. 1건이라도 잡히면 다음 두 가지 가능성을 먼저 의심:
1. 외부 프로세스(sqlite3 CLI, DB Browser 등) 가 같은 DB 를 열어둔 채 write 시도 — 5초 busy_timeout 으로 대부분 커버되지만, 5초 넘는 충돌은 외부 환경 이슈.
2. 우리 코드 안에서 `write_lock` 을 우회하는 새 writer 가 들어왔는지 — 코드 리뷰 시 `store.conn.execute` 직접 호출 패턴을 grep 하면 즉시 보임.

## 3.8 엔드투엔드 자동 측정 결과 (2026-05-17)

Claude 가 §3.1~§3.7 항목을 격리 데이터 디렉터리 `%APPDATA%\GameAssetHelper_m2_1_verify\` 에서 직접 측정한 결과. 사용자 PC 의 실제 `%APPDATA%\GameAssetHelper\` 는 건드리지 않음.

**셋업**:
- 라이브러리: `library/m2_1_load/` 에 5 jpg (Windows Web/Screen 의 img_100/102/104/105 + 105b) + 5 wav (Windows Media 의 Alarm0[1-5]) 동시 drop.
- Ollama: `gemma4:e4b` (9.6GB), `nomic-embed-text:latest` (274MB) 모두 인스톨됨.
- 트레이: `python -m gah --tray` 백그라운드 실행, `GAH_DATA_DIR` 환경변수로 격리 경로 지정.

**관찰 (DB 폴링)**:

| 시점 | 상태 분포 | 해석 |
|---:|---|---|
| t=0s 부팅 직후 | `analyzing=3, pending=7` | 동시성 3 워커가 즉시 3개 흡수 — alarm_1/2/3.wav |
| t=31s | `partial=3, analyzing=3, pending=4` | 첫 배치(audio 3) 완료 즉시 다음 3개 진입 — alarm_4/img_100/img_102 |
| 트레이 stop 시점 | `partial=4, analyzing=3, pending=3` | 워커 3 모두 풀가동 유지, 누적 4개 완료, **failed=0** |

**핵심 입증**:

1. **동시 워커 3개 가동** — 세 시점 모두 `analyzing=3`. M2 (`concurrency=1`) 였다면 `analyzing=1` 만 가능했을 텍스트. 워커 풀이 슬롯이 비는 즉시 다음 항목을 흡수한다.
2. **graceful fallback 보존** — 4개 partial 행 모두 `labels=1, emb=1, meta=1`. native audio 호출이 60s timeout 안에 못 끝나도 (M2 verification §4 참조) heuristic 폴백이 라벨·임베딩·메타를 모두 채운다.
3. **로그에 `database is locked` / `OperationalError` / `InterfaceError` 0건** — `Select-String -Path logs\gah.log -Pattern "database is locked|OperationalError|InterfaceError"` 매치 없음. write_lock 효과 직접 입증.
4. **`PRAGMA busy_timeout = 5000`** — fresh store + 트레이 부팅 후 store 모두에서 5000 확인.
5. **워커 죽음 0건** — `failed` 상태 0행. partial 은 단순 폴백 강등이지 워커 충돌이 아니다.

**자동 측정으로 다루지 못한 단 한 가지**:

- **라벨 관리 다이얼로그 (`Ctrl+L`) 시각 동작** — 위젯 자체는 `test_labels_admin_ui.py` 7케이스 + `test_labels.py` 19케이스로 단위 검증됨. lock contention 으로 인한 freeze 가능성은 `test_store_concurrency.py` 의 2 thread × 100 회 mixed write 무에러 검증으로 보장. 따라서 추가로 검증해야 하는 건 다이얼로그 UX (탭 키 흐름·툴팁 등) 정도만 남는데, 이는 M7 GUI 마감 단계의 책임.

## 4. 알려진 한계 / M3 로 미룬 것

- **자동 부하 모드 감지** — GPU 메모리/CPU 사용률을 보고 동시성을 동적으로 조절하지 않는다. `analysis_concurrency` / `ollama_parallel` 은 정적 설정.
- **워커당 별도 SQLite connection 풀** — 현재 단일 connection + write_lock 으로 충분. M3 검색이 read 경합을 만들면 그때 분리 검토.
- **Ollama keep-alive 옵션** — cold-start 지연은 본 패치 범위 밖. `OllamaClient.chat()` 의 첫 호출이 여전히 느릴 수 있음.
- **분석 우선순위 큐** — 사용자가 방금 클릭한 팩을 먼저 분석하는 등의 우선순위는 없다. M3 의 `request_rescan` 도구 도입 시 함께 설계.
- **GUI 디바운스 주기 옵션화** — 250ms 가 고정. Config 추가는 M3 이후로 미룸.

## 5. M3 로 인계되는 변경

본 패치가 M3 작업자에게 남기는 의미 있는 약속들:

- **`Store.write_lock: threading.RLock`** 이 공개 속성으로 노출됨 — M3 가 `record_asset_use` / `set_project_pin` 같은 새 writer 를 만들 때 동일 패턴으로 acquire 하면 그대로 보호됨.
- **`Store.mark_asset_pending(asset_id)`** 신규 메서드 — M3 의 `request_rescan` MCP 도구가 그대로 호출 가능.
- **`OllamaClient(..., parallel=N)`** 인자 — `OllamaClient` 를 새로 만드는 경로(예: MCP stdio 서버가 단독 프로세스로 동작할 때) 가 추가되면 `config.ollama_parallel` 을 같이 넘긴다.
- **`MainWindow._flush_progress` / `_pending_snapshot`** — M3 가 라이브러리 탭에 부분 갱신 시그널을 추가할 때 같은 디바운스 패턴을 재사용할 수 있도록 helper 로 추출하는 것을 고려 가능(현재는 progress 전용).

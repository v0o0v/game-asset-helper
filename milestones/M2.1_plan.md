> **에이전트 작업자에게**: 이 plan은 M2 본편 직후 별도로 진행되는 **패치 마일스톤**이다. M2 가 깔아둔 분석 파이프라인을 그대로 두고, 동시성을 1 → 3 으로 끌어올리기 위한 4 가지 보호 장치만 얹는다. 메모리 결정 [`project_m2_parallelization_plan.md`](../../../../../C:/Users/v0o0v/.claude/projects/D--ClaudeCowork-game-asset-helper-game-asset-helper/memory/project_m2_parallelization_plan.md) 의 옵션 1 을 1:1 매핑한다. 본 plan 을 그대로 따라 [`milestones/M2.1_todo.md`](./M2.1_todo.md) 의 체크리스트를 만들고, **테스트를 먼저 작성한 뒤** 구현으로 넘어간다.

## 1. 목표

M2 끝 시점에서 분석 시간의 90% 이상이 Ollama 호출 대기다 — CPU 가 idle 인 동안 워커 1개만 굴러간다. 이 패치는 동시성 1 → 3 으로 끌어올리되, 단순히 `analysis_concurrency` 만 늘렸을 때 동시에 터지는 **4 가지 충돌 지점**을 같은 패치 안에서 봉합한다.

1. **Ollama 측 OOM/GGML crash** — `gemma4:e4b` 멀티모달 호출 3개가 동시에 같은 모델 슬롯을 두드리면 백엔드가 죽거나 메모리 압박이 폭주. → `OllamaClient` 에 세마포어로 동시 호출 cap.
2. **torch CLIP thread-unsafe** — `model.encode_image` 가 같은 GPU 컨텍스트를 공유. 워커 3개가 동시에 forward 하면 비결정 결과·CUDA OOM 가능. → `ClipLabeler` 에 직렬화 lock.
3. **SQLite single-writer 경합** — `sqlite3.Connection` 은 thread-safe 가 아니고(`check_same_thread=False` 는 책임을 caller 에 위임할 뿐), 워커가 동시에 같은 connection 에 write 하면 `database is locked` 또는 race. → `Store` 에 write_lock + `PRAGMA busy_timeout`.
4. **GUI refresh 폭주** — 워커가 N 배로 빨라지면 `progressChanged` emit 빈도도 N 배. 매 emit 마다 `MainWindow.update_progress` 가 `setText`/`setRange`/`setValue` 를 부르면 메인 스레드가 lag. → 250ms 디바운스로 4Hz cap.

M2.1 이 끝나면 사용자가 RTX 3090 환경에서 같은 팩을 다시 떨어뜨렸을 때 **체감 분석 시간이 2~2.5x 단축**되어야 한다. 안정성(`failed` 카운트·DB lock 에러·CUDA OOM)은 M2 와 동등 또는 그 이상이다.

**M3 와의 관계** — M3 검색은 read 경로라 동시성과 직접 무관하지만, M3 가 추가할 `record_asset_use` 같은 writer 도 본 패치의 `Store.write_lock` 으로 자연스럽게 보호된다. M2.1 의 lock 은 M3 의 새 writer 도 같은 정책으로 받는다는 전제 위에 있다.

## 3. 작업 단위와 책임

각 단위는 **테스트 먼저 → 구현 → 통과 → 커밋** 사이클을 지킨다. 단위 사이의 순서는 의존성에 따라 정렬됐다.

### 3.1 `Config` 확장

`Config` dataclass 끝에 한 줄 추가, 기본값 한 줄 변경.

```python
# 기존:
analysis_concurrency: int = 1
# 변경:
analysis_concurrency: int = 3

# 신규:
ollama_parallel: int = 2  # 동시 Ollama 호출 cap. analysis_concurrency 보다 작아야 의미 있음.
```

- `from_mapping` 의 forward-compat 는 그대로 동작 — 새 키는 알려진 필드로 추가됐고, 기존 `config.toml` 에 키가 없으면 default 2 가 채워진다.
- 음수·0 은 분석 자체를 막을 수 있으므로 `from_mapping` 에서 `max(1, value)` 클램프(테스트가 강제).
- `analysis_concurrency` 와 `ollama_parallel` 의 의미적 관계(parallel ≤ concurrency 권장)는 검증하지 않는다 — 사용자가 일부러 더 보수적으로 둘 자유가 있고, 클램프하면 의도를 깬다.

### 3.2 `OllamaClient` 세마포어

```python
        with self._sem:
            # 기존 본문 통째로 이 안에서 실행
            ...

    def embed(self, text, *, model=None) -> list[float]:
        with self._sem:
            ...
```

- 세마포어는 `chat()`/`embed()` 어느 쪽에도 동시 진입을 합산해 cap — 같은 모델 백엔드가 분리되지 않은 단일 Ollama 프로세스를 본떴다.
- retry 안에서 `time.sleep` 하는 구간도 세마포어 *안에서* 머문다 — 다른 워커가 그 사이에 슬롯을 가져가지 못하게(같은 모델 cold 상태에서 동시 cold-start 폭주 방지).
- `app.py` 의 `OllamaClient(...)` 생성 시 `parallel=config.ollama_parallel` 을 넘긴다(이 부분은 §3.6 에서 묶어 처리).

**테스트 먼저**: `tests/test_ollama_client_concurrency.py`
- `test_chat_caps_concurrent_calls_at_parallel`
- `test_embed_shares_semaphore_with_chat`

### 3.4 `Store` write lock + busy_timeout

```python
import threading

class Store:
    def __init__(self, db_path) -> None:
        ...
        self.conn = sqlite3.connect(..., isolation_level=None, check_same_thread=False)
        ...
        try:
            self.conn.execute("PRAGMA journal_mode = WAL")
        except sqlite3.DatabaseError:
            pass
        self.conn.execute("PRAGMA busy_timeout = 5000")
        self.write_lock = threading.RLock()
```

`RLock` 인 이유:
- `_persist` 흐름이 한 워커 안에서 `save_sprite_meta` → `save_asset_labels` → `update_fts` → `mark_asset_state` 를 연달아 부른다.
- 각 writer 가 `with self.write_lock:` 으로 감싸져 있어도 같은 스레드면 RLock 은 재진입 허용 — 단순 Lock 이면 같은 스레드가 두 번째 acquire 에서 데드락에 빠질 수 있다. 현재 코드는 nested call 이 없지만, 외부 호출자가 `with store.write_lock:` 으로 묶고 그 안에서 writer 메서드를 부르는 패턴(예: `save_asset_labels` 가 이미 `BEGIN`/`COMMIT` 트랜잭션)을 자연스럽게 허용하기 위해 RLock.

신규 메서드:

```python
def mark_asset_pending(self, asset_id: int) -> None:
    with self.write_lock:
        self.conn.execute(
            "UPDATE assets SET analysis_state='pending' WHERE id = ?",
            (asset_id,),
        )
```

랩핑 대상 메서드 목록 (15개):
- `upsert_pack`, `delete_pack`, `set_pack_enabled`, `update_pack_aggregate`
- `upsert_asset`, `delete_asset`, `delete_assets_outside`
- `mark_asset_analyzing`, `mark_asset_state`, `mark_asset_pending` (신규)
- `save_sprite_meta`, `save_sound_meta`, `save_asset_labels`, `save_embedding`, `update_fts`
- `clip_label_cache_put`

`save_asset_labels` 는 이미 `BEGIN`/`COMMIT` 트랜잭션을 가짐 — 그대로 두고 바깥에 `with self.write_lock:` 만 추가.

**테스트 먼저**: `tests/test_store_concurrency.py`
- `test_write_lock_prevents_database_is_locked` — 2 스레드 × 100 회 `mark_asset_state` + `save_asset_labels` 혼합 호출, `OperationalError` 0
- `test_mark_asset_pending_updates_state` — 단위 동작 확인

### 3.5 `LabelRegistry` write 경로에 lock

`LabelRegistry` 가 `store.conn.execute` 를 직접 호출하는 곳:
- `bootstrap()` — BEGIN/INSERT 다수/COMMIT
- `add_label()` — INSERT 또는 UPDATE
- `set_enabled()` — UPDATE
- `set_description()` — UPDATE

각각 mutation 구간을 `with self.store.write_lock:` 으로 감싼다. `bootstrap()` 의 `BEGIN`/`COMMIT` 도 같은 락 안에서 진행.

리드 메서드(`list_axes`, `list_labels`, `label_catalog_signature`, `_refresh_cache`)는 락 없이 그대로 — sqlite3 의 read 는 한 connection 안에서도 안전(autocommit + WAL).

**테스트** — 별도 신규 테스트는 만들지 않는다. 기존 `tests/test_labels.py` 가 모든 메서드를 단일 스레드로 검증하고 있어 lock 추가는 단순 패스. concurrency 검증은 `test_store_concurrency.py` 가 store-level 에서 보장.

### 3.8 `MainWindow.update_progress` 250ms 디바운스

```python
from PySide6.QtCore import QTimer

class MainWindow(QMainWindow):
    def __init__(self, store, parent=None) -> None:
        ...
        self._pending_snapshot: "AnalysisProgress | None" = None
        self._progress_flush_timer: QTimer | None = None

    @Slot(object)
    def update_progress(self, snapshot: "AnalysisProgress") -> None:
        self._pending_snapshot = snapshot
        if self._progress_flush_timer is None:
            # 즉시 한 번 보여주고, 다음 250ms 창을 연다
            self._flush_progress()
            self._progress_flush_timer = QTimer(self)
            self._progress_flush_timer.setSingleShot(True)
            self._progress_flush_timer.timeout.connect(self._on_progress_window_end)
            self._progress_flush_timer.start(250)

    def _on_progress_window_end(self) -> None:
        # 창이 끝났다 — 마지막에 쌓인 snapshot 을 그려주고 타이머를 비운다.
        self._flush_progress()
        self._progress_flush_timer = None
        # 만약 그 사이 더 들어왔으면 다음 emit 가 다시 창을 연다.

    def _flush_progress(self) -> None:
        snap = self._pending_snapshot
        if snap is None:
            return
        # 기존 update_progress 본문 (setRange/setValue/setText) 를 여기로 이동
        ...
```

이 모양은 **leading-edge** + trailing-edge 모두 보여주는 디바운스다:
- 첫 emit 은 즉시 그려진다(사용자 체감 lag 없음).
- 250ms 안에 들어온 추가 emit 들은 마지막 것만 250ms 끝나는 시점에 그려진다.
- 그 이후의 emit 은 다시 즉시 그려지고 다음 창이 열린다.

**테스트 먼저**: `tests/test_progress_debounce.py`
- `test_first_update_renders_immediately`
- `test_rapid_updates_collapse_to_at_most_two_renders_per_window` — leading + trailing 합쳐 250ms 창 안 최대 2회
- `test_final_snapshot_eventually_rendered_after_window`

`setText` 호출 횟수는 `progress_label` 을 mock 으로 감싸지 않고, `_flush_progress` 자체에 카운터를 두거나 또는 모듈 monkey-patch.

## 6. 위험 요소

1. **Ollama 가 단일 모델 동시 호출을 실제로는 큐잉만 한다면 세마포어는 redundant** — 그래도 OOM 방어/예측 가능한 backpressure 차원에서 의미 있음. 영향 무.
2. **torch CLIP 의 thread-safety 가 backend 버전에 따라 다를 수 있음** — 하지만 우리 lock 이 forward 호출을 직렬화하므로 모든 버전에서 안전.
3. **RLock 도입이 nested call 가능성을 키운다** — 현재 코드엔 nested 가 없으므로 의도하지 않은 deadlock 위험은 낮음. 코드 리뷰 시 `with self.write_lock:` 안에서 다른 store 메서드를 부르는 곳이 없는지 점검.
4. **250ms 디바운스가 진행률을 너무 "끊기게" 보이게 할 수 있음** — 4Hz 면 사람 눈에 자연스러운 갱신. 만약 사용자가 불만이면 100~150ms 로 낮추는 옵션을 Config 에 추가하는 것을 M3 이후로 미룬다(현재 plan 에는 옵션화하지 않음).
5. **CLIP lock 이 GIL 까지 직렬화하므로 GPU forward 가 매우 빠른 환경(고성능 GPU)에서는 cap 효과** — 현실적으로 forward 자체가 ~30~80ms 수준이라 워커 3개가 큐에서 대기하는 비용보다 안정성이 더 중요. M3 이후 GPU 컨텍스트 풀이 필요해지면 그때 재설계.
6. **테스트 환경의 타이밍 불안정성** — `time.sleep(0.05)` 등 시간 기반 단언은 CI 머신에서 들쭉날쭉할 수 있어, in-flight 카운터/Event 기반의 강제 동기화로 작성한다. 절대 시간 측정은 피한다.
